# v4-4 提示词头部与规则原文输出迭代

## 修改目标

本次迭代只调整 v4-4 的 prompt 生成形态，覆盖 `10 matchResume`、`9-1 processResume`、`9-2 resumeFix`。v4-4 继续沿用 runtime `fetchAction` + function template 的模式，不引入 LLM/OpenAI 调用，不改变 v4-1/v4-2/v4-3 的行为。

## 头部结构

原先 v4-4 将执行角色、约束、Action 描述和触发条件都放在 `## 你的任务` 中。本次拆成两个章节：

- `## 你的角色`：描述执行智能体身份、判断限制、输出要求、规则处理顺序、`step_N_result` 与 `final_output` 的维护方式。
- `## 你的任务`：描述当前 Action 的任务背景、已触发事件、Action 描述、提交/触发条件原文。

## 删除内容

删除头部低价值的 `Action 信息` 块：

```text
Action 信息：
- ID: ...
- 名称: ...
- 分类: ...
- 执行者: ...
```

删除每条规则下的低价值元信息：

```text
- 严重性: ...
- 执行者: ...
- 规则来源: ...
- 适用客户: ...
```

规则标题、`适用条件原文`、`规则原文`、必要时的 `补充描述原文` 继续保留。

## 保留约束

- 规则仍采用填空式原文引用。
- `submissionCriteria` 继续完整输出为 `适用条件原文`。
- `standardizedLogicRule` 继续完整输出为 `规则原文`。
- 当 `description` 与 `standardizedLogicRule` 不同时，继续输出 `补充描述原文`，避免遗漏原文信息。
- 不添加 `How to evaluate this rule`、`What to do when this rule fires`、`How to verify before moving on` 等 LLM 改写式提示。

- v4-4 预览不包含 `Action 信息：`。
- v4-4 规则块不包含 `- 严重性:`、`- 执行者:`、`- 规则来源:`、`- 适用客户:`。
- v4-4 预览仍包含 `适用条件原文:`、`规则原文:`，以及必要时的 `补充描述原文:`。
- `9-2 resumeFix` 无规则步骤仍显示“本步骤没有业务规则，按步骤描述执行。”

## 第二轮精简

本轮继续减少模板噪音，让 v4-4 更接近“原文填空式”结构：

- 业务原文直接放在对应标题下，不再使用 `text` 代码块围栏。
- `提交/触发条件原文`、`步骤前置条件原文`、`完成条件原文`、`适用条件原文`、`规则原文`、`补充描述原文` 分别简化为 `提交/触发条件`、`步骤前置条件`、`完成条件`、`适用条件`、`规则`、`补充描述`。
- 删除每个步骤里的 `### 本步骤中间结果` 和 `### 本步骤执行方式`。
- 删除 `════════` 分割线和规则之间的 `---` 分隔符。
- 删除每条规则下的 `填空要求`，将命中、阻断、通知和写入要求统一集中到 `## 你的角色` 中。
- JSON 输出结构仍保留 `json` 代码块，因为它不是业务规则原文，且有助于保持结构可读。

- v4-4 预览不包含业务原文字段周围的 `text` 代码块围栏。
- v4-4 预览不包含 `### 本步骤中间结果`、`### 本步骤执行方式`、`填空要求:`。

- `9-2 resumeFix` 无规则步骤仍显示“本步骤没有业务规则，按步骤描述执行。”

## 第三轮精简

本轮继续收敛 step 内部结构，并为 `10 matchResume` 补充更贴近实际调用的运行时输入示例：

- 删除每个 `## Step n` 下的 `### 本步骤目标` 整段；步骤目标说明由 `## 执行步骤总览` 统一承载。
- `10 matchResume` 在未显式传入 `runtimeInput` 时，`## 运行时输入` 默认展示一组 JD 与候选人简历示例。
- `9-1 processResume`、`9-2 resumeFix` 继续保留 `{{RUNTIME_INPUT}}` 占位符；如果任一 action 在 runtime 显式传入 `runtimeInput`，仍优先展示调用方传入内容。

## 第四轮收敛

本轮继续调整头部章节、action 10 输入结构和 step/rule 层级：

- `## 你的角色` 改为 `## 角色`，只保留执行智能体身份描述。
- 原 `重要约束：` 升级为独立的 `## 重要约束` 章节，集中承载执行顺序、原文判断、`step_N_result`、`final_output`、阻断和通知写入要求。
- `## 你的任务` 改为 `## 任务`，并重写为更具体的模板任务描述：读取运行时输入，严格按照 actionSteps 顺序执行，并在每个 Step 内严格按照 rules 出现顺序判断。
- 删除 `## 任务` 中的 `提交/触发条件`。
- `10 matchResume` 的默认运行时输入从 `const` 示例改为分层结构：`client`、`招聘岗位 (Job_Requisition)`、`候选人简历 (Resume)`。
- 每条规则标题改为 step 内顺序编号格式，如 `#### 1. [规则 10-25] ...`。
- 删除 `### 本步骤完成条件`，将 step 完成、阻断返回和继续下一步的要求集中到 `## 重要约束`。
